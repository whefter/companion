# Chat Webhooks

Chat webhooks let your agents respond to mentions and messages on external platforms like Linear, GitHub, Slack, and Discord. When someone mentions your agent in a Linear issue comment, the agent starts a session, processes the request, and posts its response back to the thread — supporting full multi-turn conversations.

## How it works

1. An external platform sends a webhook when someone mentions your agent
2. The Companion matches the mention to a configured agent
3. A new session is created and the agent processes the message
4. The agent's response is posted back to the platform thread
5. Follow-up messages in the same thread continue the conversation in the same session

## Setup: Linear

### 1. Get your Linear credentials

You need two values from Linear:

| Credential | Where to find it |
|---|---|
| **API key** | [Linear Settings > API](https://linear.app/settings/api) — create a personal or workspace key |
| **Webhook secret** | [Linear Settings > Webhooks](https://linear.app/settings/api/webhooks) — create a webhook pointing to your Companion's URL |

### 2. Set environment variables

Add these to your Companion server's environment:

```bash
LINEAR_API_KEY=lin_api_...
LINEAR_WEBHOOK_SECRET=lin_wh_...
```

The Companion checks for these on startup. If both are present, the Linear chat adapter is initialized automatically.

### 3. Configure the Linear webhook

In Linear's webhook settings, create a webhook pointing to:

```
https://your-companion:3456/api/chat/webhooks/linear
```

<Warning>
The webhook endpoint (`/api/chat/webhooks/linear`) is intentionally **unauthenticated** — Linear verifies requests using HMAC signatures with your webhook secret. Do not put this endpoint behind additional auth that would block Linear's requests.
</Warning>

### 4. Create a chat-enabled agent

1. Go to **Agents** in the sidebar
2. Create or edit an agent
3. In the **Triggers** section, toggle **Chat** on
4. Click **Add platform** and select **linear**
5. Configure:
   - **Mention pattern** (optional): A regex that filters which messages trigger this agent. For example, `@reviewer` only triggers when the message contains "@reviewer". Leave empty to respond to all mentions.
   - **Auto-subscribe**: When enabled, the agent automatically subscribes to the thread for follow-up messages, enabling multi-turn conversations.
6. Save the agent

### 5. Test it

In a Linear issue, post a comment mentioning your bot. The Companion picks up the webhook, finds the matching agent, starts a session, and posts the response back as a comment on the issue.

## Mention patterns

The **mention pattern** field accepts a regular expression that the incoming message must match for the agent to respond. This is useful when you have multiple agents listening on the same platform.

| Pattern | Matches |
|---|---|
| `@reviewer` | Messages containing "@reviewer" |
| `@bot\b` | Messages containing "@bot" as a whole word |
| `(review\|audit)` | Messages containing "review" or "audit" |
| *(empty)* | All messages (no filter) |

<Note>
Mention patterns are case-insensitive and tested against the first 1,000 characters of the message for safety.
</Note>

## Multi-turn conversations

When **auto-subscribe** is enabled for a chat platform binding:

1. On the first mention, the agent starts a new session and subscribes to the thread
2. Follow-up messages in the same thread are injected into the existing session as user messages
3. The agent continues the conversation with full context from the session history
4. Thread state (session ID, agent ID) is persisted so conversations survive server restarts

If a follow-up message arrives on a thread that has no active session (e.g., the session was closed), a new session is created automatically.

## Cloud relay

If your Companion runs behind a firewall or NAT (no public URL), you can use the **cloud relay** — a Cloudflare Worker that forwards webhooks to your local Companion over a WebSocket tunnel.

### 1. Deploy the relay worker

The relay worker source is in `relay/` at the repository root. Deploy it to Cloudflare:

```bash
cd relay
npm install
npx wrangler deploy
```

Set the `RELAY_SECRET` environment variable in your Cloudflare Worker settings:

```bash
npx wrangler secret put RELAY_SECRET
```

### 2. Connect the Companion

Add these environment variables to your Companion server:

```bash
COMPANION_RELAY_URL=https://your-relay.your-worker.workers.dev
COMPANION_RELAY_SECRET=your-shared-secret
```

On startup, the Companion opens a WebSocket connection to the relay. When an external platform sends a webhook to the relay (`POST https://your-relay.workers.dev/webhooks/linear`), the relay forwards it to the Companion over the WebSocket, waits for the response, and returns it to the platform.

### 3. Update the Linear webhook URL

Point your Linear webhook to the relay instead of the Companion:

```
https://your-relay.your-worker.workers.dev/webhooks/linear
```

### Relay health check

Check if the Companion is connected to the relay:

```bash
curl https://your-relay.your-worker.workers.dev/health
# {"connected": true}
```

<Note>
The relay uses per-isolate in-memory state. For low-traffic, single-tenant deployments this works well. For production-grade high-availability, consider migrating to Cloudflare Durable Objects so WebSocket state is shared across Worker isolates.
</Note>

## Monitoring: Runs page

The **Runs** page (`#/runs` in the sidebar) shows execution history across all agents, including chat-triggered runs.

Each execution entry shows:
- **Agent name** and trigger type (manual, webhook, schedule, chat)
- **Status**: Running, Success, or Error
- **Timing**: When it started, how long it ran
- **Session link**: Click to jump to the session output

Use the filters at the top to narrow by agent, trigger type, or status.

## Verify the setup

After configuring everything:

1. Check that platforms are detected:

```bash
curl http://localhost:3456/api/chat/platforms \
  -H "Authorization: Bearer YOUR_TOKEN"
# {"platforms": ["linear"]}
```

2. The `/api/chat/platforms` endpoint requires authentication. If it returns an empty array, check that your `LINEAR_API_KEY` and `LINEAR_WEBHOOK_SECRET` environment variables are set correctly and restart the Companion.

3. Post a test comment in a Linear issue and watch the **Runs** page for a new execution.

## Permissions

Chat-triggered agent sessions always run with **bypass permissions** — no interactive prompts for tool approval. This is necessary because chat sessions are headless (no browser UI for permission prompts).

For Claude Code sessions, this sets `--permission-mode bypassPermissions`. For Codex sessions, the approval policy is set to `never` with full sandbox access.

<Warning>
Since chat-triggered agents run with full permissions, ensure the agent's prompt and allowed tools are appropriately scoped. Consider using the **Allowed Tools** restriction in the agent's Advanced settings to limit what the agent can do.
</Warning>
